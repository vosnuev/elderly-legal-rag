# Demo 3. 데이터 출처, 전처리, 최종 QA 평가 계획

## 이 파트의 핵심 메시지

이 RAG 시스템은 문서를 임의로 넣은 데모가 아니라, 법령/조례 데이터를 수집하고 전처리한 뒤 사람이 검수 가능한 그래프 후보 생성 파이프라인으로 연결한 구조다.

핵심은 세 가지다.

1. 데이터는 `law.go.kr` Open API 기반으로 수집했다.
2. API 원본 JSON은 RAG 입력 효율을 위해 TOON 형식으로 전처리했다.
3. 최종 질의응답은 MCP로 노출된 RAG 그래프를 조회하고, 환각 방지 테스트로 검증한다.

## 데이터 출처와 수집 근거

| 구분 | 근거 코드/디렉토리 | 설명 |
| --- | --- | --- |
| 법령 수집 | `rag/code_reference/collect.py` | `law.go.kr` DRF API의 `lawSearch.do`, `lawService.do`를 사용해 법령 목록과 본문 JSON을 수집한다. |
| 조례 수집 | `rag/code_reference/collect_ordinance.py` | `law.go.kr` DRF API의 `target=ordin` 검색을 사용해 고령자/노인복지/노인일자리 등 키워드 기반 조례 JSON을 수집한다. |
| 법령 전처리 | `rag/code_reference/preprocess_law.py` | 원본 JSON에서 조문 단위 문서 구조를 추출한다. |
| 조례 전처리 | `rag/code_reference/preprocess_ordinance.py` | 조례 JSON에서 조문 내용과 메타데이터를 추출한다. |
| 최종 RAG 입력 데이터 | `rag/RAG_PREPROCESSED_DATA/rag_datas` | 실제 업로드/검증에 사용한 `.toon` 전처리 데이터가 들어 있다. |

발표 멘트:

> 데이터 수집은 코드로 재현 가능하게 관리했습니다. 법령은 국가법령정보 API에서 법령 목록과 본문을 JSON으로 받고, 조례는 같은 API의 조례 검색을 사용했습니다. 이후 원본 JSON을 그대로 LLM에 넣지 않고, RAG 입력에 적합한 TOON 형식으로 변환했습니다.

## 수집 대상

| 데이터 그룹 | 대상 |
| --- | --- |
| 노동/고용 관련 법령 | 근로기준법, 최저임금법, 고용보험법, 근로자퇴직급여 보장법, 고용상 연령차별금지 및 고령자고용촉진에 관한 법률 |
| 고령자 고용 관련 조례 | 부산광역시 수영구, 울산광역시, 인천광역시, 전북특별자치도 전주시의 고령자 경비원/공동주택 노동자 관련 조례 |

제외한 데이터도 있다. 복지, 기초연금, 노인돌봄, 경로당, 노인여가, 주거취약계층, 정보취약계층처럼 이번 RAG 목적과 직접 관련성이 낮은 문서는 최종 TOON 데이터에서 제외했다.

## 원본 형식과 전처리 형식

| 단계 | 형식 | 특징 |
| --- | --- | --- |
| API 원본 | JSON | 법령/조례의 계층 구조와 메타데이터가 포함되어 있으나, RAG 입력으로는 중복 키와 구조 노이즈가 많다. |
| 중간 처리 | 조문 단위 document | 조문 내용, 법령명, 조문 번호, 제목 등 검색과 청킹에 필요한 필드를 추출한다. |
| 최종 입력 | TOON (`.toon`) | 구조는 유지하되 JSON보다 토큰을 적게 쓰는 텍스트 표현으로 변환한다. |

TOON은 JSON처럼 구조화된 데이터를 표현하지만, 반복되는 따옴표, 중괄호, 키 이름 부담을 줄여 LLM 컨텍스트에 더 많은 실제 본문을 넣기 위한 형식이다.

발표 멘트:

> 원본 JSON은 사람이 보기에는 명확하지만 LLM 입력으로는 비효율적입니다. 같은 내용을 표현해도 JSON 문법 토큰이 많이 들어가기 때문입니다. 그래서 최종 입력은 TOON으로 바꿨고, 이 덕분에 같은 컨텍스트 길이 안에서 더 많은 법령 조문을 다룰 수 있었습니다.

## TOON 전처리 효과

토큰 수 비교는 `rag/RAG_PREPROCESSED_DATA/README.md` 기준이다. 토크나이저는 `cl100k_base`를 사용했다.

| 항목 | 원본 | TOON | 절감 |
| --- | ---: | ---: | ---: |
| 전체 토큰 | 320,991 | 189,997 | 130,994 |
| 전체 바이트 | 917,767 | 463,900 | 453,867 |

중요한 점은 절감률을 문서별 평균으로 계산하지 않았다는 것이다. 문서마다 원본 길이가 크게 다르기 때문에, 전체 원본 토큰 합계와 전체 TOON 토큰 합계를 기준으로 계산했다.

```txt
토큰 절감률 = (전체 원본 토큰 - 전체 TOON 토큰) / 전체 원본 토큰
           = (320,991 - 189,997) / 320,991
           = 40.81%
```

| 데이터 | 원본 토큰 | TOON 토큰 | 절감률 |
| --- | ---: | ---: | ---: |
| 고용보험법 | 115,253 | 69,580 | 39.63% |
| 고용상 연령차별금지 및 고령자고용촉진에 관한 법률 | 29,248 | 17,299 | 40.85% |
| 근로기준법 | 88,011 | 44,267 | 49.70% |
| 근로자퇴직급여 보장법 | 53,479 | 36,058 | 32.58% |
| 최저임금법 | 20,772 | 10,303 | 50.40% |
| 고령자 조례 | 14,228 | 12,490 | 12.22% |

발표 멘트:

> 전체 기준으로 약 40.81%의 토큰을 줄였습니다. 이 수치는 문서별 절감률의 평균이 아니라 전체 토큰 합계 기준입니다. 긴 문서와 짧은 문서의 영향이 다르기 때문에, 실제 시스템 비용과 컨텍스트 효율을 보려면 전체 합계 기준으로 보는 것이 맞습니다.

## RAG 구축 플로우와 연결

이 데이터는 단순히 벡터 DB에 넣고 끝나는 구조가 아니다.

1. 사용자가 `.toon` 문서를 업로드한다.
2. RAG backend가 문서를 DB에 저장하고 ingestion job을 만든다.
3. construction graph가 문서를 chunk로 나누고 embedding을 저장한다.
4. graph candidate agent가 chunk 간 관계 후보를 만든다.
5. 사용자는 review queue에서 후보 edge를 승인하거나 거절한다.
6. 승인된 후보는 실제 그래프 edge로 저장된다.
7. 사용자의 review note는 memory update agent를 통해 agent memory에 반영된다.
8. 이후 candidate agent는 업데이트된 memory를 컨텍스트로 받아 더 나은 후보를 생성한다.

발표 멘트:

> 여기서 중요한 점은 RAG 그래프를 자동으로 바로 확정하지 않는다는 것입니다. 에이전트는 edge candidate를 만들고, 사용자가 review queue에서 승인한 관계만 실제 그래프 edge가 됩니다. 또 사용자가 남긴 리뷰 노트는 memory layer로 업데이트되어 다음 candidate generation에 반영됩니다. 즉, 시스템이 사용자와 함께 작업하면서 점진적으로 기준을 학습하는 구조입니다.

## 최종 질의응답 시스템 플로우

최종 질의응답은 문서 처리 파이프라인과 분리해서 생각할 수 있다. 처리된 그래프는 MCP 서버를 통해 외부 에이전트나 백엔드에서 조회할 수 있는 지식 인터페이스가 된다.

```txt
사용자 질문
  -> Streamlit / frontend
  -> main backend
  -> RAG MCP server
  -> Memgraph graph / vector index / text index
  -> 검색된 근거 chunk와 relationship
  -> backend answer generation
  -> 사용자 응답
```

발표 멘트:

> 문서 업로드와 그래프 구축은 비동기 파이프라인으로 처리하고, 실제 질의응답 시점에는 MCP 서버를 통해 구축된 그래프를 조회합니다. 이렇게 하면 RAG backend는 지식 그래프를 제공하는 도구 서버가 되고, 실제 응답 생성 에이전트는 필요한 시점에 graph, vector, text search tool을 조합해서 사용할 수 있습니다.

2. 근거가 없을 때 지어내지 않고 제한적으로 답변하는가.

5. agent memory가 candidate generation 기준 개선에 영향을 주는가.

## 발표 마무리 멘트

> 이번 RAG 파이프라인의 목표는 단순 검색이 아니라 검수 가능한 지식 그래프 구축입니다. 데이터는 국가법령정보 API 기반으로 수집했고, TOON 전처리로 토큰 효율을 높였습니다. 이후 에이전트가 chunk, embedding, relationship candidate를 만들고, 사용자는 review queue에서 관계를 확정합니다. 마지막으로 review note가 memory layer에 반영되어 다음 그래프 구축 기준을 개선합니다. 내일은 리모트에 있는 모델 테스트 결과와 360개 환각 방지 테스트 결과를 병합해서 최종 QA 보고서로 정리할 예정입니다.
