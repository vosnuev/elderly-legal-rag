## 논의 배경

데이터 수집 범위가 너무 넓어져서 사용할 데이터 종류와 RAG 파이프라인 입력 기준을 먼저 구체화해야 한다는 의견이 나왔다. 특히 RAG 구축 자체만 한 사람이 맡기에는 부담이 크므로, 먼저 수집 대상 데이터를 주제별로 좁히고 이후 세부 역할을 나누는 방향으로 논의했다.

## 핵심 결정 사항

1. 데이터 탐색과 1차 수집은 2026-05-26 화요일까지 진행한다.
2. 이번 단계에서는 역할을 세부적으로 나누기보다, 전원이 법률 RAG에 필요한 자료를 먼저 찾고 정리한다.
3. 3차 산출물의 주제 범위는 `고령층을 위한 법률 RAG Agent`로 제한한다.
4. 3차 산출물에는 법률 데이터만 우선 포함한다. 복지, 지역별 지원, 일자리, 음성/방언 대응은 이번 범위에서 제외한다.
5. 복지 데이터와 지역별 지원 정보는 4차 프로젝트에서 법률 RAG와 함께 확장하는 방향으로 가져간다.
6. 챗봇 형태는 고령층 사용성을 고려하면 어려움이 있을 수 있으므로, 4차 프로젝트에서는 키오스크 또는 말로 대화하는 형태의 인터페이스 확장을 검토한다.

## 데이터 범위 정리

### 3차 우선 범위

- 고령층과 직접 관련된 법령 문서

- 출처와 조항 정보를 함께 남길 수 있는 법령 원문 또는 법령 PDF

### 후보 법령 자료

- `노인복지법`
- `고용상 연령차별금지 및 고령자고용촉진에 관한 법률`
- 기존 논의에서 언급된 고령자고용촉진 관련 법령

### 3차 범위에서 제외하고 4차로 넘길 항목

- 지역별 복지 지원 정책
- 광역시/특별시 단위의 고령층 지원 정보
- 공공기관/민간기관 지원 서비스 분류
- 고령층 대상 채용/일자리 정보
- 방언, 어르신 표현, 음성 인식 보조
- 키오스크형 UI 또는 음성 대화형 UX

## 작업 디렉터리

GWS의 `sk_그룹3` 공유 디렉터리 아래 `Rag` 폴더를 기준 작업 공간으로 사용한다.

- 기준 경로: `sk_그룹3/Rag/`
- 현재 확인된 법령 수집 폴더: `sk_그룹3/Rag/국가법령정보/`
- 폴더 링크: <https://drive.google.com/drive/folders/1VSJmMQqe_RZU0-SF4QE0K7HjPIb5EMxy>

추가 수집 자료는 `Rag` 하위에 데이터셋 단위로 폴더를 하나씩 만들어 올린다.

권장 구조:

```txt
sk_그룹3/
└── Rag/
    ├── 국가법령정보/
    ├── 노인복지법/
    │   ├── raw/
    │   └── README.md
    └── 고령자고용촉진법/
        ├── raw/
        └── README.md
```

각 데이터셋 폴더에는 최소한 다음 내용을 남긴다.

- `raw/`: 원본 파일, 원본 PDF, 원문 HTML, API 응답 원본
- `README.md`: 출처 URL, 수집일, 수집 방식, 담당자, 포함 법령, 파이프라인 입력 메모

수동 처리 산출물 폴더를 따로 만들지는 않는다. 3차 구현에서는 원본 데이터를 RAG 파이프라인에 연결하고, 필요한 추출/분할/임베딩 처리는 코드 파이프라인에서 수행한다.

## 사용성 관련 논의

고령층 사용자를 고려하면 단순 챗봇은 사용이 어려울 수 있다는 의견이 있었다. 다만 3차 산출물에서는 법률 RAG의 검색과 답변 품질을 먼저 만든다. 최종 서비스 형태는 4차 프로젝트에서 키오스크처럼 보이는 화면 또는 말로 대화하는 방식으로 발전시키는 것이 적절하다는 방향성이 나왔다.

음성 모델 자체를 파인튜닝하는 것은 GPU 부담이 크기 때문에 현 단계에서는 현실성이 낮다고 봤다. 대신 방언 데이터셋이나 지역별 표현 데이터를 수집해 LLM에 힌트로 제공하는 방식은 가능성이 있다고 정리했다.
